# SDG Connection Test — Privacy Notice

This document describes what data the SDG Connection Test server collects,
how long it is retained, and what it is used for. It applies to the server
component of the project. The client component does not collect anything;
see [`client/README.md`](../client/README.md) and
[`docs/TRANSPARENCY.md`](TRANSPARENCY.md) for the full client-side audit.

If you are a player running the client against the SDG-operated test
server, this is the data the operator may collect about your connection.
If you are an operator running your own deployment, this is the data your
deployment will collect about clients that connect to it. In either case
you should read this document.

## What is collected

The server's session log
([`server/sessionLog.js`](../server/sessionLog.js)) records one JSON line
per logical "session" — a tuple of (source IP, client-generated nonce). A
new session begins when a client emits a previously-unseen nonce; an
existing session is updated when packets continue arriving with the same
(IP, nonce) pair. Each line contains:

| Field            | Example                              | Notes                                                                |
|------------------|--------------------------------------|----------------------------------------------------------------------|
| `ts`             | `2025-09-12T18:42:00.123Z`           | Time the session was flushed to disk                                 |
| `srcIp`          | `203.0.113.42`                       | Source IPv4 address of the client                                    |
| `asn`            | `21928`                              | Autonomous System Number, looked up offline (best-effort)            |
| `ispLabel`       | `T-Mobile USA`                       | Human ISP label, looked up offline                                   |
| `nonceHex`       | `0011223344556677`                   | 8-byte random value generated by the client per run                  |
| `firstSeen`      | `2025-09-12T18:41:30.000Z`           | First packet of the session                                          |
| `lastSeen`       | `2025-09-12T18:42:00.123Z`           | Last packet of the session                                           |
| `durationMs`     | `30123`                              | Wall-clock duration of the session                                   |
| `counts`         | `{ "udp:27015": 50, "udp:27016": 600 }` | Per-(proto:port) probe count                                       |
| `a2sQueries`     | `1`                                  | Number of A2S_INFO queries                                           |
| `streamsServed`  | `1`                                  | Number of game-shape sustained tests served                          |
| `rateLimited`    | `0`                                  | Number of times this client tripped a rate limit                     |

The server does **not** collect:

- The contents of any probe packet beyond its size and type
- Anything the client did NOT send (no browser fingerprinting, no
  device introspection, no geolocation calls — see
  [`docs/TRANSPARENCY.md`](TRANSPARENCY.md))
- Any data tied to a Steam account, an in-game identity, or any other
  third-party identifier
- Any data from clients that did not connect (no scanning, no probing)

The ASN/ISP lookup is performed offline against a small bundled file
([`server/data/ip-to-asn.tsv`](../server/data/ip-to-asn.tsv)). No
network request is made during lookup.

## How long it is retained

The log file is hard-capped at 50 MB. When that cap is exceeded the log
is rotated:

```
sessions.log      → sessions.log.old   (overwriting any previous .old)
sessions.log      ← (empty, new file)
```

This means the server keeps **at most two generations of logs at any
time**, with a hard ceiling of approximately 100 MB on disk. The previous
`.old` file is deleted on each rotation. There is no archival, no off-host
shipping, and no long-term store.

For deployments expecting heavy traffic, the operator should either
increase the rotation cap or — preferably — wire up a host-side cron that
archives `sessions.log.old` before the next rotation overwrites it. See
[`docs/DEPLOY.md`](DEPLOY.md) §14 for an example. Any such
archival is the operator's responsibility and is OUTSIDE the scope of
this notice.

## How it is used

The session log exists for two narrow purposes:

1. **Field-test correlation.** When a client run is captured to a JSON
   report, the server-side log lets the operator check what the *server*
   observed for the same nonce, so loss/jitter measured client-side can
   be compared with what reached the server. Without this, a one-sided
   measurement is hard to interpret. The matched ASN label makes the
   per-carrier comparison the project was built for legible.

2. **Abuse detection.** Sustained per-IP rate-limit hits are counted so
   the operator can see if a single source IP is flooding the server. The
   server's rate limiter handles the actual throttling; this field is
   purely informational.

The data is **not** shared, exported, sold, or correlated with any other
data set. It does not leave the host the server runs on except via
operator-initiated archival.

## Legal posture

The IP address of a connecting client may, in some jurisdictions, be
considered personal data (notably under GDPR in the EU and the UK Data
Protection Act). If you operate a public-facing deployment of this
software and clients in those jurisdictions can reach it, you are the
data controller for the resulting log and are responsible for compliance
(lawful basis, retention, deletion-on-request, etc.). The defaults
described above — short retention via rotation, no sharing, no offsite
storage, narrow purpose — are intended to support a "legitimate
interests" basis under GDPR Article 6(1)(f), but the legal analysis is
the operator's, not the project's.

If you are a player and you want a copy of, or the deletion of, log
entries that pertain to your IP, contact the operator of the deployment
you connected to. The SDG-operated reference deployment will honor such
requests within the limits imposed by the rotation policy (i.e. the
operator cannot delete from a generation that has already been
overwritten — once `sessions.log.old` is rotated out, it is gone).

## Operator responsibilities

If you stand up your own deployment, please:

- Either link to this file from your deployment's user-facing
  documentation, or write your own equivalent. Players running the
  diagnostic should know what they are agreeing to.
- Do not ship the log to a SaaS, log aggregator, or third-party
  analytics tool without amending this notice and disclosing the
  destination.
- Keep the server on a wired or business connection. Running it on a
  residential or mobile carrier (T-Mobile 5G Home, Verizon hotspot,
  etc.) can violate the carrier's residential ToS and put your
  account at risk; that risk is yours, not the project's.
