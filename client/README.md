# SDG Connection Test — Client

A small diagnostic tool for players having trouble connecting to SDG's Space
Engineers / Torch servers. It tests TCP and UDP connectivity to an SDG-operated
test server on every port SE, Torch, and Steam actually use, and reports
whether packets are getting through and how cleanly.

**Nothing about your computer, your Steam install, or your game is sent
anywhere.** This README describes exactly what the tool does and does not
do, and tells you how to verify that for yourself.

---

## TL;DR — how to run it

1. Install [Node.js 20 or later](https://nodejs.org/en/download) (LTS is fine).
2. Download `sdg-connection-test.zip` from <https://sdg.knowledgeondemand.net> and unzip it.
3. Open a terminal (PowerShell, cmd, or bash) in the unzipped folder.
4. Run:

   ```
   node client/client.js --host <server-given-by-SDG>
   ```

5. Review the "what this will do" summary the tool prints.
6. Press `y` to proceed.
7. Wait about 90-120 seconds.
8. Read the results table. If SDG asked for a JSON report, re-run with
   `--json report.json` and send them the file.

Full options:

```
node client/client.js --help
```

## What it sends

Every outbound packet is one of exactly two shapes:

1. **An SDG Connection Test probe** — a 36-byte binary header (plus optional
   zero padding up to 1400 bytes for the MTU test) containing:
   - The four ASCII bytes `SDGT`
   - A version byte (`0x01`)
   - A type byte (`0x01` probe, `0x03` stream-begin, `0x04` stream-stop)
   - Two reserved zero bytes
   - An **8-byte random session nonce** generated by `crypto.randomBytes`
     when the client starts. The nonce is printed to the console at startup,
     so you can filter a Wireshark capture on it.
   - A 4-byte sequence number
   - A client-local timestamp (nanoseconds since process start; **not a
     wall clock and not a system identifier**)
   - Eight zero bytes (server-timestamp field on reply; zero on probes)
   - Zero padding, if any

2. **A standard Steam A2S_INFO request** on UDP port 27015 only:

   ```
   FF FF FF FF 54 "Source Engine Query\0"
   ```

   This is the exact same byte sequence the Steam server browser sends to
   every listed game server, documented publicly by Valve at
   <https://developer.valvesoftware.com/wiki/Server_queries>.

Both packet shapes are described byte-by-byte in [../docs/PROTOCOL.md](../docs/PROTOCOL.md).

## What it does NOT send

- **No username, hostname, MAC address, or machine ID.** None of these are
  read anywhere in the code.
- **No file contents.** The tool never opens any file for reading except
  the two sibling `.js` files it needs to run (`../shared/ports.js` and
  `../shared/protocol.js`).
- **No environment variables.** None are read.
- **No game data, no Steam data, no save files, no Torch configs.**
- **No telemetry.** There is no phone-home. The only network destinations
  are the `--host` you typed on the command line and — only if you pass
  `--real-server` — the second host you typed.
- **No auto-update, no dynamic code loading.** The tool contains zero calls
  to `eval`, zero `require`s other than Node.js built-ins and the two
  sibling files mentioned above. Search `client.js` for `require(` and
  `net.`, `dgram.`, `fs.` to see every I/O point.

## How to verify all of the above

Choose one or more of:

1. **Read the code.** `client/client.js` is the entire client — a single
   file, around 500 lines, with comments. Read it top to bottom. It should
   not take more than ten minutes.
2. **Run a packet capture.** Open Wireshark on your active NIC, start the
   tool, note the 16-hex-character session nonce it prints, then filter
   your capture with `frame contains <nonce-bytes>` or simply
   `host <server>`. Every single packet the tool sends should be one of the
   shapes described above.
3. **Run it offline.** With no network interface, the tool will fail every
   test and exit. It will not "fall back" to any other behavior.
4. **Read the sibling files.** `../shared/ports.js` is the list of ports
   probed; `../shared/protocol.js` is the binary packet format. Both are
   fully commented and together are under 200 lines.

## What the tests do

| Test | Where | What it proves |
| --- | --- | --- |
| Reachability | Every (proto, port) in the table | One packet round-trips within 2 seconds |
| Loss rate | Every UDP port | 50 packets at 10 pps, % received |
| Latency stats | Every UDP port | min / avg / p95 / max / stddev RTT |
| MTU / large packet | Every UDP port | A 1400-byte packet still round-trips |
| Steam A2S query | UDP 27015 | A real Source-engine query gets a real reply — the same thing the Steam server browser does |
| Game-shape sustained | UDP 27016 | The server pushes ~60 pps, 200-400 byte packets for 10 seconds. Measures how many you receive and whether there are any gaps larger than 250 ms (the signature of ISP shaping/throttling) |
| `--real-server` A2S (opt-in) | Your Torch server | Sends an A2S query to the actual server you are trying to play on, for side-by-side comparison |

## Reading the results

- **`proto` / `port` / `cat` / `purpose`** — what was tested.
- **`reach`** — `OK` if a single round-trip happened within 2 seconds,
  `FAIL` otherwise.
- **`rtt(ms)`** — average RTT for UDP (from the loss test) or the single
  round-trip time for TCP.
- **`loss%`** — percentage of probes that did not come back (UDP only).
- **`mtu1400`** — whether a 1400-byte probe still came back. `FAIL` here
  with a pass on the small probe points at an ISP that is clamping MTU or
  mangling large UDP packets, which is a known T-Mobile 5G Home pattern.
- **Game-shape line** — look for `[THROTTLING SIGNAL]`. That appears when
  any inter-packet gap in the 10-second sustained test exceeds 250 ms,
  which is well above normal network jitter and suggests active traffic
  shaping.
- **Steam A2S line** — if this fails but TCP 443 works, you have a Steam
  UDP problem specifically.

## Command-line options

```
--host <addr>            SDG server to probe (required).
--ports <p1,p2,...>      Only test these port numbers.
--no-sustained           Skip the 10-second UDP 27016 game-shape test.
--no-a2s                 Skip the Steam A2S query test.
--real-server <h:p>      Also send A2S to your real Torch server.
--duration <seconds>     Override the sustained test duration (default 10).
--json <file>            Write a full JSON report to <file>.
--yes, -y                Skip the confirmation prompt.
--help, -h               Show this help.
```
